01. 아이템 선정

포트폴리오의 시작은 팬덤 소재 선정입니다. 좋아하는 팬덤보다 결과물의 완성도를 높일 수 있는 팬덤을 선택해야 합니다. 선정 명분이 분명하고, 팬덤 모바일웹으로 구현했을 때 시각적 임팩트가 큰 팬덤이 면접관의 시선을 더 끌지 않을까요?

  • Before & After의 결과가 명확할 수 있는가
  • 현재 모바일웹의 문제점이 명확해 선정 명분이 생기는가
  • 이미지, 영상, 공간, 제품, 라이프스타일 등 비주얼 소스가 풍부한가
  • 하단 탭, 팬 피드, 멤버 카드, 투표/응원, 라이브 일정, 굿즈/이벤트, 알림, 채팅창 인터랙션이 자연스럽게 어울리는가
  • 최상급 퀄리티의 레퍼런스 자료가 충분한가

최상급 포트폴리오가 되는 팬덤 소재 기준

좋은 소재는 “좋아하는 대상”이 아니라 모바일 기능으로 설계할 팬 활동이 많은 대상입니다. 멤버, 콘텐츠, 이벤트, 라이브, 커뮤니티, 채팅, 응원 행동이 자연스럽게 연결될수록 포트폴리오 완성도가 높아집니다.

  • 합격 소재: 멤버/캐릭터/선수처럼 프로필화할 대상이 있고, 팬 피드·투표·라이브·이벤트·채팅으로 확장 가능한 팬덤
  • 보류 소재: 비주얼은 좋지만 팬 행동이 단순 감상에 그쳐 앱 기능으로 확장하기 어려운 소재
  • 탈락 소재: 이미지 소스가 부족하고, 팬끼리 상호작용할 이유나 채팅창을 만들 명분이 약한 소재
  • 최상급 판단: “왜 이 팬덤은 모바일웹이어야 하는가?”와 “왜 채팅창이 필요한가?”를 한 문장으로 설명할 수 있어야 한다.

06. 참고 레퍼런스 제시

레퍼런스 사이트는 프로젝트 퀄리티를 좌우하는 핵심 요소입니다. 제작 방향에 맞는 최상급 레퍼런스 사이트 7개를 제안하고, 사용자가 직접 기준 레퍼런스를 선택합니다. 적합한 사이트가 없다면 후보를 다시 추천하며, 최종 선정된 사이트는 다음 단계의 디자인 시스템 추출 기준으로 사용합니다.

  • 기준 레퍼런스 후보 7개를 먼저 제시한다.

  • 사용자가 후보 중 1개를 선택할 수 있도록 장단점을 명확히 정리한다.
  • 선택된 1개 사이트만 다음 단계에서 디자인 시스템 추출 대상으로 사용한다.

레퍼런스 후보 선정 원칙

레퍼런스는 사용자가 충분히 비교·판단할 수 있도록 최소 7개의 후보를 제시하며, 모든 레퍼런스는 실제 확인 가능한 URL을 함께 제공합니다.

사용자는 제시된 후보를 검토한 후 최종 기준 레퍼런스 1개를 직접 선정합니다. 선정된 기준 레퍼런스는 프로젝트의 전체 레이아웃, 타이포그래피, 그리드 시스템, 콘텐츠 구성의 기준이 되며, 이후 디자인 방향성 역시 이를 중심으로 전개합니다.

기준 레퍼런스 선정은 프로젝트의 최종 비주얼 퀄리티를 결정하는 핵심 과정입니다. 따라서 사용자가 원하는 방향성과 수준에 부합하는 레퍼런스를 찾을 때까지 후보 탐색, 비교, 교체 과정을 반복적으로 진행합니다. 충분한 검토 없이 레퍼런스를 확정하지 않으며, 사용자가 납득할 수 있는 기준과 근거를 바탕으로 최종 레퍼런스를 선정합니다.

서브 레퍼런스는 기본적으로 제공하지 않으며, 사용자가 추가 요청한 경우에만 히어로 섹션, 카드 UI, 모션, CTA, 인터랙션 등 특정 요소에 한하여 별도로 제안합니다. 단, 서브 레퍼런스는 부분 요소 참고용으로만 활용하며 전체 디자인 구조의 기준은 항상 기준 레퍼런스를 유지합니다.

3. selector, CSS rule, computed style, screenshot 측정 위치 중 하나 이상의 근거가 있어야 함
4. 확인하지 못한 값은 "추정"이 아니라 "확인 필요"로 남겨야 함

10. 와이어프레임 제작

Figma MCP를 연결한 후, 선정한 레퍼런스 사이트의 메인(index) 페이지 전체 캡처를 Figma에 배치합니다. 이후 07단계에서 정리한 디자인 시스템과 08단계에서 등록한 Figma Local Style 및 Token을 확인하고, 09단계에서 작성한 정보구조도(IA)를 기준으로 와이어프레임을 설계합니다.

이때 레퍼런스 사이트는 레이아웃과 구성 방식을 참고하기 위한 기준으로 활용하며, 실제 와이어프레임의 페이지 구성과 섹션 순서는 반드시 09 정보구조도에 정의된 구조를 따라 제작합니다. 단순히 레퍼런스를 복제하는 것이 아니라, 우리 프로젝트에 맞게 재구성된 전체 페이지 구조를 시각화하는 과정입니다.

와이어프레임은 텍스트 설계서가 아닌 Figma 화면으로 직접 제작하며, 이 단계에서도 07에서 추출한 디자인 시스템과 08에서 등록한 스타일을 적용합니다. 이를 통해 그리드, 타이포그래피 계층, 여백, 이미지 비율 등이 실제 디자인 단계에서도 자연스럽게 이어질 수 있는지 미리 검증합니다.

와이어프레임 최상급 합격 기준

최상급 와이어프레임은 회색 박스가 아니라, 최종 디자인과 구현의 뼈대가 되는 모바일 앱 설계도입니다. 각 화면에서 무엇을 보여주고, 사용자가 무엇을 누르며, 어떤 상태가 바뀌는지까지 보여야 합니다.

  • 낮은 퀄리티: Home에 Hero, Card, CTA만 있고 팬덤앱 기능 화면이 없다.
  • 보통 퀄리티: 멤버/피드/투표/채팅 화면은 있으나 상태값, 빈 상태, 오류 상태가 없다.
  • 최상급 퀄리티: Home, Members, Feed, Vote, Live, Notification, Chat List, Chat Room, My가 430px 프레임으로 있고, 각 화면에 기본/활성/완료/빈 상태가 표시된다.
  • 채팅창 합격 기준: 말풍선 배치뿐 아니라 입력 전, 입력 중, 전송 중, 전송 완료, 읽음, 실패, 신고/차단 상태가 모두 설계되어야 한다.

기준 레퍼런스 캡처 방법

기준 레퍼런스 캡처는 사용자가 직접 하는 작업이 아니라, AI 에이전트가 먼저 해결해야 하는 작업입니다. AI는 Playwright, 브라우저 캡처, PDF 저장, 분할 캡처, 이미지 변환 등 가능한 방법을 순서대로 시도해 기준 레퍼런스 index 페이지를 캡처하고 Figma Reference Frame에 배치해야 합니다.

AI 우선 캡처 원칙
1. AI가 Playwright fullPage screenshot 또는 브라우저 자동화 캡처를 먼저 시도
2. 실패하면 AI가 브라우저 기본 캡처, PDF 저장, 분할 캡처, HTML 저장 후 로컬 캡처를 순서대로 시도
3. 그래도 어렵다면 AI가 가능한 확장 프로그램, 외부 캡처 도구, 온라인 캡처 서비스 사용 가능성을 확인
4. 설치, 로그인, 유료 서비스, 권한 승인이 필요한 경우에만 사용자에게 가능 여부를 묻기
5. 사용자가 직접 캡처 파일을 만들어 주는 것을 기본 전제로 하지 않기

캡처가 확보되지 않았다면 CSS만 보고 와이어프레임을 확정하지 않습니다. CSS/DOM 분석은 보조 근거로만 사용하고, 캡처 이미지나 PDF를 Figma Reference Frame에 배치한 뒤 진행합니다.

   - 해외 팬덤 사이트, 에디토리얼 디자인, 매거진 레이아웃처럼 설명문 자체를 시각적 콘텐츠로 활용한다.
   - 설명문은 콘텐츠 밀도와 레이아웃에 맞게 배치하며, 화면이 단조롭거나 썰렁해 보이지 않도록 구성한다.
   - 단순 제목과 한두 줄 설명만 배치하여 섹션을 마무리하지 않는다.
12. Hero를 먼저 20년차 시니어 디자이너 3명이 승인할 수 있는 최상급 퀄리티로 완성한 뒤, 같은 디자인 밀도로 전체 페이지까지 확장
13. 반복 검수, 수정, 보완을 계속 진행해 레이아웃, 타이포, 이미지, 여백, 컬러, 카피의 수준을 최상급 디자인 퀄리티까지 끌어올릴 것
14. 단순 카드 리스트, 기능 소개 섹션, Feature Grid 반복만으로 페이지를 완성하지 말고, 각 페이지는 팬덤의 세계관과 핵심 메시지가 전달되는 스토리텔링 흐름으로 구성할 것
15. 페이지 안에는 팬덤을 대표하는 Mobile(430px) 비주얼 섹션을 최소 1~2개 이상 반드시 포함할 것
   - 해당 섹션은 max-width: 430px 비주얼 이미지를 중심으로 구성
   - 팬덤 서사, 감성, 세계관이 전달되는 강한 타이포그래피와 카피를 함께 배치
   - 일반 콘텐츠 섹션보다 더 큰 스케일과 여백을 사용해 스크롤 중 강한 인상을 줄 것
16. 단순 정보 전달만 하지 말고 이미지, 타이포그래피, 레이아웃을 활용해 팬덤 스토리가 자연스럽게 이어지는 흐름을 구성할 것
17. 시안 디자인 단계에서는 제목과 설명만 있는 와이어프레임형 카드 사용을 금지한다. 모든 카드와 콘텐츠 영역은 실제 서비스 수준의 비주얼, 이미지, 아이콘, 데이터 요소 등을 포함한 완성형 디자인으로 제작할 것
18. 팬덤앱 핵심 화면은 실제 서비스처럼 완성한다.
   - 멤버 프로필: 멤버 이미지, 역할, 대표 콘텐츠, 팬 반응
   - 팬 피드: 게시글, 이미지, 댓글 수, 좋아요/응원 상태
   - 응원 투표: 진행률, 랭킹, 참여 완료 상태
   - 라이브 일정: 일정 카드, 라이브 상태, 알림 신청
   - 굿즈/이벤트: 상품/이벤트 카드, 마감 상태, CTA
   - 알림: 새 소식, 라이브 알림, 투표 결과, 배지
   - 채팅창: 채팅 목록, 채팅 상세, 메시지 입력, 전송/읽음 상태, 시스템 메시지, 안전 안내
19. 채팅창 디자인은 단순 데모가 아니라 포트폴리오에서 핵심 기능으로 설명할 수 있는 수준으로 완성한다.
   - 팬덤 무드에 맞는 말풍선 스타일
   - 날짜 구분선과 시간 표시
   - 내 메시지/상대 메시지 구분
   - 이모지, 응원 배지, 이미지 메시지 예시
   - 입력창, 첨부, 이모지, 전송 버튼
   - 신고/차단/운영 안내 등 커뮤니티 안전 UX

12-2. 디자인 검수

12-1에서 제작한 최종 디자인 결과물을 Goal 프롬프트의 최종 디자인 제작 기준 1~22번에 따라 체크리스트로 검수합니다. 각 항목의 충족 여부를 확인하고, 미흡한 부분은 Figma 안에서 직접 수정·보완합니다.

검수는 한 번으로 종료하지 않고, 사용자가 최종 승인할 때까지 반복합니다. 사용자가 [다음단계진행]이라고 답하면 현재 디자인을 완료 상태로 간주하고 다음 단계로 이동합니다. 사용자가 [다시]라고 답하면 Goal 프롬프트를 다시 적용하여 1~22번 기준에 따라 재검수하고, 부족한 부분을 Figma 안에서 계속 수정·보완하여 디자인 퀄리티를 업그레이드합니다.

14. 팬덤 모바일웹 React 구현

취업을 위한 팬덤 모바일웹 포트폴리오는 정적인 랜딩페이지가 아니라, 적절한 인터랙티브 요소를 활용해 팬덤 경험을 전달하고 면접관의 시선을 사로잡을 수 있도록 설계되어야 합니다.

14-1. 기본 인터랙티브 구현

구현된 React 모바일웹에 기본 스크롤 이벤트, 섹션 등장, 호버, 메뉴 이동, CTA 피드백 등 필수 인터랙션을 적용합니다.

채팅창 UI 스펙 예시

채팅창은 팬덤앱 포트폴리오의 핵심 차별점입니다. 단순 입력창이 아니라 상태와 안전 UX가 포함된 실제 서비스 화면으로 구현합니다.

  • Chat List: 방 이름, 아바타, 최근 메시지, 시간, 안읽음 배지, 라이브 상태
  • Chat Room Header: 뒤로가기, 방 이름, 참여자/상태, 메뉴 버튼
  • Message Area: 날짜 구분선, 내/상대 말풍선, 이미지/이모지/응원 배지 메시지
  • Message Status: sending, sent, read, failed 상태를 시각적으로 구분
  • Chat Input: 텍스트 입력, 첨부, 이모지, 전송 버튼, 빈 입력 비활성화
  • Safety UX: 신고, 차단, 운영 안내, 부적절 메시지 신고 CTA
14-1. 기본 인터랙티브 구현

너는 기본 인터랙션을 정교하게 구현하는 팬덤 모바일웹 React 구현 전문가야.

현재 구현된 React 팬덤 모바일웹을 취업을 위한 UIUX 모바일 포트폴리오 수준의 팬덤 모바일웹으로 보완해줘.
goal은 "430px 이하 모바일에서 최상급 고퀄리티 팬덤앱 경험처럼 보이는 React/TypeScript/Tailwind CSS 구현"으로 설정하고, 목표에 도달할 때까지 검토·수정·보완을 반복해줘.

보완할 인터랙션:
1. 히어로 진입 모션
2. 스크롤 섹션 등장 애니메이션
3. 이미지 또는 카드 호버 반응
4. 모션 타이포 또는 숫자 카운트
5. 메뉴 이동과 현재 섹션 표시
6. CTA 버튼 피드백
7. 모바일 메뉴

주의:
- 과한 애니메이션보다 팬덤 무드에 맞는 리듬을 우선
- 스크롤 이벤트는 필요한 경우에만 사용하고, 성능을 해치지 않도록 IntersectionObserver, requestAnimationFrame, CSS transition을 우선 활용
- prefers-reduced-motion 대응
- 모바일에서 텍스트와 버튼이 겹치지 않게 확인
- TypeScript 오류, React key warning, 콘솔 오류, 모바일 성능 저하가 없게 확인
- 기존 13 React 바이브코딩에서 완성한 레이아웃과 카피를 임의로 바꾸지 말 것

완료 후 어떤 인터랙션을 어디에 적용했는지 표로 정리하고, goal 기준 반복 검토·수정·보완 완료 여부를 표시해줘.

14-2. 사용자 요청 팬덤 모바일웹 React 구현

사용자가 추가로 요청한 특정 모션이나 인터랙션을 선택 섹션 단위로 구현하고, 브라우저에서 바로 QA합니다.

팬덤 모바일웹 React 구현 요청 시 사용자가 준비할 프롬프트 작성요령

원하는 인터랙션은 단순히 “움직이게 해주세요”가 아니라, 어디에 적용되는지, 어떤 조건에서 시작되는지, 어떻게 변화하는지, 어디에서 끝나는지를 기준으로 작성합니다.

가능하면 레퍼런스 URL, 영상, GIF를 함께 제공하고 모바일에서 동일하게 유지할지 여부도 명확히 작성합니다.

요청 형식

1. 인터랙션 목표: [사용자가 어떤 경험을 하길 원하는지]
2. 적용 위치: [페이지 / 섹션 / 요소명]
3. 참고 레퍼런스: [URL 또는 설명]
4. 시작 상태: [처음 화면에서 보이는 상태]
5. 작동 조건: [스크롤 / 호버 / 클릭 / 로딩 / 드래그 등]
6. 변화 과정: [움직임 순서와 변화 방식]
7. 종료 상태: [최종적으로 보이는 모습]
8. 모션 스타일: [부드럽게 / 강렬하게 / 고급스럽게 / 에디토리얼하게 / 미니멀하게 등]
9. 모바일 기준: [유지 / 축소 / 제거]
10. 성능 기준: [60fps 유지, 과도한 애니메이션 금지 등]
11. 금지 사항: [텍스트 겹침, 레이아웃 깨짐, 과한 모션, 성능 저하 등]

예시

1. 인터랙션 목표: 사용자가 Hero를 스크롤할 때 팬덤의 몰입감을 느끼게 한다.
2. 적용 위치: Home / Hero / Hero Image, Hero Title
3. 참고 레퍼런스: Apple Vision Pro Hero
4. 시작 상태: 제품 이미지는 중앙에 크게 배치하고, 타이틀은 opacity 0 상태로 시작한다.
5. 작동 조건: 스크롤
6. 변화 과정: 이미지가 천천히 확대되고, 타이틀이 아래에서 위로 등장하며, 스크롤 진행에 따라 이미지가 화면 상단으로 이동한다.
7. 종료 상태: 이미지가 상단에 고정되고 다음 섹션 콘텐츠가 노출된다.
8. 모바일 기준: 이미지 확대 효과만 유지하고 타이틀 애니메이션은 단순화한다.
9. 무드: 고급스럽고 몰입감 있게
10. 금지 사항: 과한 패럴랙스, 텍스트 가독성 저하, 프레임 드랍
