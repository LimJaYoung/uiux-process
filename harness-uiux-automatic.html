01. Claude Code 설치

Claude Code는 터미널에서 실행되는 AI 코딩 에이전트입니다. 웹 채팅과 달리 실제 프로젝트 폴더를 읽고, 파일을 만들고, 수정하고, 명령어까지 실행할 수 있습니다.

  • Node.js 18 이상 설치 확인
  • Claude Code 설치와 로그인
  • 프로젝트 폴더에서 실행
  • 기본 명령어와 첫 작업 확인

02. 하네스란

하네스는 AI가 혼자서도 잘 일할 수 있도록 맥락, 제한, 작업 흐름, 검증 기준을 갖춘 작업 환경입니다.

단순히 프롬프트를 잘 쓰는 것과 다릅니다. AI가 만들고, 평가하고, 고치는 기준을 미리 설계해서 결과물이 기준을 통과할 때까지 반복하게 만드는 방식입니다.

  • Planner — 아이디어를 PRD, IA, 유저플로우로 펼칩니다.
  • Generator — UI 구조, 디자인 시스템, 실제 코드를 만듭니다.
  • Evaluator — 기준표로 결과물을 채점하고 수정 항목을 골라냅니다.
  • Refiner — Fix 항목을 반영하고 다시 검수합니다.

PDF 핵심: 작업 환경 설계

첨부 자료에서는 하네스를 특정 스킬이나 에이전트 조합이 아니라, AI가 어느 맥락에서 무엇을 하고 무엇을 하면 안 되는지 알 수 있게 만드는 환경 설계로 설명합니다.

Level 1

Prompt Engineering

"이렇게 말해봐" 질문을 잘하면 답이 좋아지는 단계입니다.

Level 2

Context Engineering

"이런 배경지식을 줘봐" 맥락을 알려주면 AI가 상황을 더 잘 이해합니다.

Level 3

Harness Engineering

"환경 자체를 설계하자" AI가 혼자서도 반복 수행할 수 있는 작업 환경을 만듭니다.

맥락 (Context)

프로젝트 목적, 사용자, 기존 문서, 우선순위를 알려줘 AI가 무엇을 알아야 하는지 정합니다.

제한 (Constraints)

금지 행동, 코드 스타일, 파일 수정 범위, 테스트 조건처럼 하면 안 되는 일을 명확히 둡니다.

작업 흐름 (Workflow)

계획, 실행, 검증, 개선의 순서를 정해 AI가 임의로 건너뛰지 않게 만듭니다.

검증 (Verification)

완료 조건과 체크 기준을 먼저 정해 결과물이 믿을 수 있는지 확인합니다.

하네스 엔지니어링 6단계

PDF의 전체 흐름은 하네스를 한 번에 만드는 것이 아니라, 구조를 만들고 맥락을 넣은 뒤 계획, 실행, 검증, 개선을 반복하는 방식으로 정리됩니다.

01

구조

소스, 문서, 테스트, 설정을 하나의 프로젝트에서 관리하고 역할별 폴더를 나눕니다.

AI가 전체 맥락을 한눈에 파악합니다.
02

맥락

CLAUDE.md, 규칙, 점진적 노출로 AI가 알아야 할 배경을 단계별로 제공합니다.

아래 폴더일수록 더 좁고 우선순위 높은 규칙을 둡니다.
03

계획

작업 전에 목표, 범위, 순서, 예상 산출물을 먼저 정의합니다.

핵심 흐름은 계획 → 실행 → 검증입니다.
04

실행

계획된 범위 안에서 실제 파일을 만들고 수정합니다. 중간 산출물도 남깁니다.

한 번에 완성보다 작은 단위 실행이 안정적입니다.
05

검증

기준이 있어야 검증이 가능합니다. 완료 조건, 테스트, 빌드, 화면 기준을 먼저 정합니다.

미달이면 다시 계획부터 돌립니다.
06

개선

결과를 보고 규칙, 구조, 체크리스트를 업데이트해 다음 작업의 품질을 높입니다.

하네스는 프로젝트와 함께 점점 좋아집니다.

설정 파일과 맥락 관리

PDF에서는 설정 파일을 한 번에 모두 읽히는 긴 문서가 아니라, 범위와 우선순위를 가진 맥락 계층으로 설명합니다.

전역 규칙

~/.claude/CLAUDE.md

나만 적용 · 모든 프로젝트에 공통으로 쓰는 작업 성향과 기본 규칙

프로젝트 규칙

my-app/CLAUDE.md

팀 공유 · Git으로 관리하는 프로젝트 목적, 기술 스택, 작업 방식

주제별 규칙

.claude/rules/*.md

코드 스타일, 테스트, 보안처럼 주제별로 분리하고 glob 조건을 붙입니다.

폴더별 규칙

src/auth/CLAUDE.md

해당 폴더 작업 시에만 자동 로드되는 좁고 강한 맥락입니다.

검증 원칙: 기준이 있어야 반복된다

하네스의 검증은 "좋아 보인다"가 아니라 작업 시작 전에 합의한 완료 조건을 기준으로 합니다. 기준이 명확해야 AI가 실패를 인식하고 다시 돌 수 있습니다.

Sprint Contract

작업 전에 무엇을 만들고 어떻게 검증할지 먼저 합의합니다.

  1. 완료 조건을 먼저 정한다. 이 조건이 충족되면 끝이라고 시작 전에 합의합니다.
  2. 조건을 측정 가능하게 쓴다. "잘 되게"가 아니라 테스트 통과, 빌드 성공, 화면 기준처럼 적습니다.
  3. 미달이면 다시 돌린다. 기준이 있으니 자동 반복과 개선이 가능해집니다.

왜 UIUX 포트폴리오에 필요한가?

UIUX 포트폴리오는 결과 화면만 예쁘게 만드는 작업이 아닙니다. 문제 정의, 설계, 디자인 시스템, 구현, QA, 배포가 모두 연결되어야 합니다. 하네스는 이 과정을 하나의 반복 루프로 묶어 품질을 안정화합니다.

기준 설정 (Rules)

색상, 타이포, 여백, 접근성, 반응형, 금지 스타일을 먼저 정의합니다.

생성 (Generate)

PRD, 화면 명세, 코드, README, 배포 설정을 단계별로 만듭니다.

평가 (Evaluate)

Pass, Fix, Fail 기준으로 결과물을 검수하고 근거를 남깁니다.

재수정 (Refine)

Fix 항목만 골라 다시 수정한 뒤 재검수하고 다음 단계로 넘어갑니다.

Evaluator 평가 기준

  • Design Quality: 색상, 타이포, 레이아웃이 하나의 정체성을 만드는가?
  • Originality: AI 기본 패턴이 아니라 프로젝트 맞춤 선택이 보이는가?
  • Craft: 여백, 위계, 명도 대비, 반응형, 접근성 기준을 지키는가?
  • Functionality: 사용자가 주요 액션을 직관적으로 이해하고 수행할 수 있는가?

03. AI 에이전트 팀 구성

Claude Code의 서브에이전트는 특정 역할에 특화된 AI 팀원입니다. 복잡한 작업을 한 대화에 몰아넣지 않고, 기획·디자인·개발·QA처럼 역할별로 나누면 결과가 더 안정적입니다.

  • 각 에이전트는 독립된 컨텍스트에서 작업합니다.
  • 읽기 전용, 수정 가능 등 도구 권한을 역할별로 제한할 수 있습니다.
  • 한 번 만든 에이전트는 여러 프로젝트에서 재사용할 수 있습니다.

/agents로 에이전트 만들기

  1. Claude Code를 실행한 뒤 /agents를 입력합니다.
  2. Library 탭에서 Create new agent를 선택합니다.
  3. 저장 위치는 개인 전체 프로젝트에서 쓰려면 Personal, 현재 프로젝트 전용이면 Project를 선택합니다.
  4. Generate with Claude를 선택하고 만들고 싶은 역할을 자연어로 설명합니다.
  5. 도구 범위, 모델, 색상, 메모리 옵션을 선택한 뒤 저장합니다.

UIUX 포트폴리오용 추천 에이전트 팀

UX Planner

서비스 목표, 타깃, 문제 정의, IA, 유저플로우, 화면 목록을 정리합니다.

UI System Designer

Design.md, tokens.css, 컴포넌트 규칙, 금지 스타일을 정의합니다.

Frontend Builder

HTML/CSS/JS 또는 React 컴포넌트로 실제 작동하는 화면을 구현합니다.

QA Evaluator

접근성, 반응형, 기능, 콘솔 에러, 디자인 시스템 준수 여부를 검수합니다.

직접 파일로 작성하는 에이전트 예시

더 세밀하게 제어하려면 ~/.claude/agents/ 또는 .claude/agents/ 폴더에 마크다운 파일로 에이전트를 만들 수 있습니다. 

---
name: uiux-reviewer
description: UIUX 리뷰 에이전트. HTML/CSS/React 화면 분석 시 사용.
model: claude-sonnet-4-5
tools:
  - Read
  - Grep
  - Glob
---

당신은 UIUX 전문 리뷰어입니다.
프로젝트 파일을 읽고 다음 관점으로 피드백합니다.

- 접근성: 대비, 포커스, 키보드 탐색
- 반응형: 모바일 390px, 데스크톱 1920px 프레임
- 레이아웃: 여백, 정렬, 시각적 위계
- 디자인 시스템: 토큰, 컴포넌트, 금지 스타일 준수
- 기능 상태: 기본, 빈 상태, 로딩, 에러, 성공 피드백

파일을 직접 수정하지 말고,
문제 위치, 이유, 개선 방향, 수정 예시를 함께 제시하세요.
04 Automation Pipeline

UIUX 포트폴리오 자동화 파이프라인

첨부된 하네스 파일은 Claude Code가 기획, 설계, 디자인, 구현, 검증, 배포를 순서대로 통과하게 만드는 수업용 제작 시스템입니다. 핵심은 "한 번에 알아서 만들어줘"가 아니라, 각 단계의 전문 에이전트가 산출물을 남기고 다음 단계가 그 문서를 재료로 이어받는 구조입니다.

환경 설치 /plan /design /generate /build /review /deploy
  • 순서 강제: 이전 단계 산출물이 없으면 다음 에이전트가 멈추고 필요한 파일을 안내합니다.
  • 역할 분리: planner, architect, designer, builder, reviewer, deployer가 각자 한 단계만 책임집니다.
  • 문서 축적: 모든 판단 근거는 docs/에 저장되어 포트폴리오 과정 설명 자료가 됩니다.
  • 완성 검증: Critical 이슈가 0개일 때만 배포 단계로 넘어가는 품질 게이트를 둡니다.
Setup

0단계. 첨부 파일을 프로젝트에 배치하기

ZIP 파일의 7개 핵심 파일과 commands 폴더를 아래 구조로 넣으면 Claude Code가 세션 시작 시 CLAUDE.md를 읽고, 슬래시 명령으로 각 에이전트를 호출할 수 있습니다.

하네스

00_CLAUDE.md를 프로젝트 루트의 CLAUDE.md로 복사합니다. 전체 규칙, 단계 순서, 산출물 기준을 담습니다.

에이전트 6개

01_planner.md부터 06_deployer.md까지 번호를 제거해 .claude/agents/에 넣습니다.

명령어 6개

plan.md, design.md, generate.md, build.md, review.md, deploy.md를 .claude/commands/에 넣습니다.

산출물 폴더

docs/에는 문서가, src/에는 목업과 실제 HTML/CSS/JS 구현 파일이 생성됩니다.

# 새 포트폴리오 프로젝트 생성
mkdir my-portfolio
cd my-portfolio
mkdir -p .claude/agents .claude/commands docs src

# 하네스 파일
cp 00_CLAUDE.md CLAUDE.md

# 에이전트 파일: 번호를 제거해서 복사
cp 01_planner.md   .claude/agents/planner.md
cp 02_architect.md .claude/agents/architect.md
cp 03_designer.md  .claude/agents/designer.md
cp 04_builder.md   .claude/agents/builder.md
cp 05_reviewer.md  .claude/agents/reviewer.md
cp 06_deployer.md  .claude/agents/deployer.md

# 슬래시 명령 파일
cp commands/plan.md      .claude/commands/plan.md
cp commands/design.md    .claude/commands/design.md
cp commands/generate.md  .claude/commands/generate.md
cp commands/build.md     .claude/commands/build.md
cp commands/review.md    .claude/commands/review.md
cp commands/deploy.md    .claude/commands/deploy.md
Run Order

1단계. 슬래시 명령으로 단계별 실행하기

각 명령은 하나의 역할과 하나의 결과물을 가집니다. 중간 단계를 건너뛰면 후속 에이전트가 필요한 선행 문서를 확인하고 멈추도록 설계되어 있습니다.

01

/plan

planner가 서비스명, 타깃, 문제 정의, 핵심 기능을 질문하고 PRD를 작성합니다.

생성: docs/prd.md
02

/design

architect가 PRD를 읽고 IA, 사용자 흐름, 화면 목록, 핵심 화면 정의서를 만듭니다.

생성: docs/ia.md, userflow.md, screens.md
03

/generate

designer가 디자인 시스템, 컬러 토큰, 컴포넌트 스펙, HTML 목업 시안을 만듭니다.

생성: docs/design-spec.md, src/mockup/
04

/build

builder가 디자인 명세와 목업을 실제 반응형 HTML/CSS/JS로 구현합니다.

생성: src/index.html, src/css/, src/js/
05

/review

reviewer가 시맨틱 HTML, 토큰 사용, 반응형, 접근성, UX, 성능 기준을 검수합니다.

생성: docs/review-report.md
06

/deploy

deployer가 Critical 0개 상태를 확인하고 GitHub Pages 또는 Vercel 배포를 진행합니다.

생성: docs/deploy-info.md, 라이브 URL

Quality Gate

2단계. 단계 완료 기준 확인하기

자동화 파이프라인에서 중요한 것은 "AI가 답했다"가 아니라 "다음 단계가 읽을 파일이 생겼다"입니다. 아래 체크리스트를 기준으로 수업 중 진행 상태를 확인합니다.

/plan docs/prd.md

문제 정의, 페르소나, MoSCoW 기능 목록, KPI가 정리되어야 합니다.

/design docs/ia.md, docs/userflow.md, docs/screens.md

화면 ID, 주요 사용자 흐름, 핵심 화면 정의가 연결되어야 합니다.

/generate docs/design-spec.md, src/css/tokens.css, src/mockup/

디자인 원칙, 토큰, 대표 화면 HTML 시안이 브라우저에서 확인되어야 합니다.

/build src/index.html, src/css/main.css, src/js/main.js

완성형 반응형 포트폴리오가 375px부터 데스크톱까지 깨지지 않아야 합니다.

/review docs/review-report.md

Critical 이슈가 0개이고 수정 지시가 구체적으로 기록되어야 합니다.

/deploy docs/deploy-info.md

라이브 URL, 저장소 링크, 재배포 방법이 남아야 합니다.

Class Flow

3단계. 실습 중 막혔을 때 복구하기

문제가 생기면 코드를 먼저 고치기보다 "어느 단계 산출물이 비어 있는지"를 확인합니다. 이 파이프라인은 앞 단계 문서가 다음 단계의 입력값이 되는 구조라서, 누락 파일만 찾아도 대부분의 문제가 정리됩니다.

복구 원칙

막히면 직전 단계 산출물부터 확인합니다.

  1. /design이 멈추면 docs/prd.md가 있는지 확인합니다.
  2. /generate가 멈추면 docs/screens.md가 있는지 확인합니다.
  3. /build가 멈추면 docs/design-spec.md와 src/css/tokens.css를 확인합니다.
  4. /deploy가 멈추면 docs/review-report.md의 Critical 이슈 수를 확인합니다.
