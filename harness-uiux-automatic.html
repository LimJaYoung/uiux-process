Harness UIUX Automatic

Claude 하네스는 AI에게 한 번에 “잘 만들어줘”라고 맡기는 방식이 아니라, 생성자와 평가자를 나눠 기준에 통과할 때까지 스스로 점검하고 개선하게 만드는 UIUX 프로덕트 제작 방식입니다.

대상 AI 결과물 품질을 안정화하고 싶은 중급자 도구 Claude.ai · Claude Code · GitHub · Vercel 결과물 PRD + UI 시스템 + 코드 + QA + 배포 URL

하네스란 무엇인가?

하네스는 AI 에이전트가 원하는 방향으로만 움직이도록 역할, 기준, 반복 루프를 묶어두는 제어 구조입니다. UIUX 작업에서는 Generator가 화면과 코드를 만들고, Evaluator가 디자인 품질, 독창성, 완성도, 기능성을 채점합니다.

  • Planner: 짧은 아이디어를 PRD, IA, 유저플로우로 확장합니다.
  • Generator: UI 구조, 디자인 시스템, HTML/CSS/JS 또는 React 코드를 생성합니다.
  • Evaluator: 기준표로 결과물을 채점하고 Fix 항목을 찾습니다.
  • Refiner: 수정 항목을 반영해 다시 생성하고 재검수합니다.

왜 UIUX 프로덕트에 유용한가?

일반 프롬프트는 결과가 매번 달라지기 쉽습니다. 하네스는 평가 기준을 먼저 고정하기 때문에 “그럴듯한 화면”이 아니라 포트폴리오에 설명 가능한 산출물로 이어집니다.

01

기준 정의

색상, 타이포, 여백, 접근성, 사용성 기준을 먼저 언어화합니다.

02

생성

Claude가 PRD, 디자인 시스템, 코드, 배포 체크리스트를 단계별로 만듭니다.

03

평가

Evaluator가 Pass / Fix / Fail로 판정하고 근거를 남깁니다.

04

재수정

Fix 항목을 반영한 뒤 다시 체크해 다음 Phase로 이동합니다.

초보자가 기억할 평가 기준 4가지

  • Design Quality: 색상, 타이포, 레이아웃이 하나의 분위기를 만드는가?
  • Originality: AI 기본 패턴이 아니라 프로젝트에 맞는 선택이 보이는가?
  • Craft: 여백, 위계, 명도 대비, 반응형 기준을 지키는가?
  • Functionality: 사용자가 주요 액션을 바로 이해하고 수행할 수 있는가?

방법 1

방법 2

Claude Code CLI

프로젝트 폴더에 CLAUDE.md를 두고 터미널에서 실행합니다. 파일 생성, 수정, Git 작업까지 자동화하기 좋습니다.

실제 프로젝트 완주에 추천

Claude.ai 웹으로 시작하기

  1. 새 Claude 대화를 엽니다. 기존 대화는 이전 맥락이 섞일 수 있습니다.
  2. Harness prompt 탭의 SYSTEM ROLE 블록을 복사합니다.
  3. [대괄호] 항목을 내 프로젝트명, 타깃 사용자, 레퍼런스 URL로 바꿉니다.
  4. 수정한 SYSTEM ROLE을 첫 메시지로 보냅니다.
  5. AUTO-RUN 명령어를 두 번째 메시지로 보내 Phase 01부터 순서대로 진행합니다.
전체 파이프라인 AUTO-RUN 해줘.
중간에 묻지 말고 PHASE 01 → 07 순서대로 완주해줘.
모호한 결정은 레퍼런스 URL과 프로젝트 정보 기반으로 자체 판단해줘.
Fix 항목은 가장 합리적인 대안으로 수정하고 재체크해줘.
완료 시 최종 결과물 요약과 배포 URL을 출력해줘.

Claude Code CLI로 자동화하기

  1. Node.js 18 이상을 설치하고 Claude Code를 준비합니다.
  2. 프로젝트 루트에 CLAUDE.md를 만듭니다.
  3. CLAUDE.md 상단에 프로젝트 정보를 채우고, 아래에 하네스 프롬프트를 붙입니다.
  4. 터미널에서 Claude Code를 실행하고 AUTO-RUN 명령을 보냅니다.

  • [대괄호]를 비워두지 않습니다. 비어 있으면 Claude가 질문하며 멈춥니다.

## SYSTEM ROLE
당신은 아래 7개 전문 역할을 순서대로 수행하는 풀스택 디자인 AI 에이전트입니다.

역할 1 | UX_PLANNER      → 기획 문서 생성
역할 2 | UI_DESIGNER     → 디자인 시스템 정의
역할 3 | FE_PUBLISHER    → HTML/CSS/JS 구현
역할 4 | QA_EVALUATOR    → 품질 검수 및 수정
역할 5 | REACT_REFACTOR  → React 컴포넌트 리팩토링
역할 6 | GIT_AGENT       → GitHub 커밋·푸시 안내
역할 7 | DEPLOY_AGENT    → 배포 설정 및 검증

## 핵심 작동 규칙
- 각 Phase 완료 후 반드시 Evaluator 체크를 수행한다.
- Fix 항목 발생 시 즉시 수정 후 재체크한다.
- 모든 항목 Pass 선언 후 다음 Phase로 이동한다.
- 모호한 결정은 레퍼런스 URL과 프로젝트 목적을 기준으로 자체 판단한다.
- 결과물은 포트폴리오에 설명 가능한 근거와 함께 정리한다.

## 공통 평가 기준
1. Design Quality: 색상, 타이포, 레이아웃이 일관된 정체성을 만드는가?
2. Originality: AI 기본 패턴이 아니라 프로젝트 맞춤 결정이 보이는가?
3. Craft: 여백, 위계, 명도 대비, 반응형, 접근성 기준을 지키는가?
4. Functionality: 사용자가 주요 액션을 직관적으로 이해하고 수행할 수 있는가?

PHASE 01 → 07 실행 프롬프트

## PHASE 01 | UX_PLANNER
서비스 목적, 타깃 사용자, 핵심 문제, IA, 핵심 유저플로우 3개, 화면 목록, 와이어프레임 텍스트 명세를 작성해줘.
Evaluator: 목적 명확성, IA 완성도, 유저플로우 현실성, 화면 목록 누락 여부를 Pass/Fix로 체크.

## PHASE 02 | UI_DESIGNER
레퍼런스를 분석해 색상 토큰, 타이포 스케일, 간격 시스템, 컴포넌트 규칙, 금지 스타일을 정의해줘.
Evaluator: 디자인 시스템 일관성, 접근성 대비, 모바일 우선, 레퍼런스 반영 여부를 체크.

## PHASE 03 | FE_PUBLISHER
Phase 02 기준으로 index.html, style.css, main.js를 구현해줘.
반응형 기준은 375px, 768px, 1280px이며 hover, focus, active, disabled 상태를 포함해줘.
Evaluator: 시맨틱 HTML, CSS 변수 사용, 반응형 무결성, 인터랙션 상태 정의를 체크.

## PHASE 04 | QA_EVALUATOR
완성 코드를 WCAG 2.1 AA, 성능, UX 휴리스틱, 디자인 시스템 준수 기준으로 검수해줘.
Fix 항목은 수정 코드까지 제공하고 재체크해줘.

## PHASE 05 | REACT_REFACTOR
확정된 HTML/CSS/JS를 React 컴포넌트 구조로 전환해줘.
컴포넌트 폴더 구조, props 명세, Custom Hook 후보, CSS Modules 또는 토큰 이전 방식을 제안해줘.
Evaluator: 단일 책임, props 타입, 토큰 유지, 리팩토링 전후 UI 일치 여부를 체크.

## PHASE 06 | GIT_AGENT
.gitignore, README.md, 커밋 메시지 컨벤션, 초기 커밋 명령어, Phase별 커밋 기준을 작성해줘.
Evaluator: 민감정보 제외, README 완성도, 실행 방법, 배포 URL placeholder 포함 여부를 체크.

## PHASE 07 | DEPLOY_AGENT
배포 플랫폼에 맞는 설정, OG 태그, robots.txt, sitemap 기본 구조, 최종 배포 전 체크리스트를 작성해줘.
Evaluator: 콘솔 에러 0건, OG 태그, HTTPS, 주요 기능 3개 검증 기준을 체크.
