Claude Harness

Claude 하네스는 AI를 생성자와 평가자로 분리해, 기준을 통과할 때까지 스스로 반복 검증하게 만드는 구조입니다.
"한 번에 완성"이 아니라, 루프를 돌며 품질을 높이는 방식으로 UIUX 산출물을 만들어냅니다.

01 대상 AI 결과물이 매번 들쭉날쭉한 게 고민인 중급자 02 도구 Claude.ai · Claude Code · GitHub · Vercel 03 결과물 PRD부터 배포 URL까지기획·디자인·코드·QA 한 사이클 완성

하네스란 무엇인가?

하네스는 AI가 엉뚱한 방향으로 흐르지 않도록, 에이전트에게 역할·제작 기준·반복 루프를 하나로 묶어 제어하는 구조입니다.
UIUX에서는 만드는 AI와 검토하는 AI를 분리해, 기준을 통과할 때까지 반복합니다.

단순히 프롬프트를 잘 쓰는 것과 다릅니다. AI가 스스로 만들고 → 채점하고 → 고치는 흐름을 설계하는 방식으로, 프로덕트의 퀄리티를 높입니다

  • Planner — 아이디어 한 줄을 PRD·IA·유저플로우로 펼칩니다
  • Generator — UI 구조와 디자인 시스템, 실제 동작하는 코드를 만듭니다
  • Evaluator — 기준표로 결과물을 채점하고 고칠 것만 골라냅니다
  • Refiner — Fix 항목을 반영해 다시 만들고, 다시 검수합니다

왜 UIUX 프로덕트에 유용한가?

하네스는 기준을 먼저 잡고, 그 기준을 통과할 때까지 반복합니다.

프롬프트만으로는 얻기 어려운 일관성과 완성도, 하네스가 만드는 차이입니다.

Planner (기획)

색상·타이포·여백·접근성을 기준표로 먼저 만듭니다. 기준이 없으면 평가도 없습니다.

Generator (생성)

PRD부터 코드·배포까지 단계별로 만들어냅니다.

Evaluator (평가)

Pass·Fix·Fail로 판정하고, 왜 그런지 근거까지 기록합니다.

Refiner (재수정)

Fix 항목만 골라 다시 만들고, 통과하면 다음 단계로 넘어갑니다.

Evaluator의 평가기준

  • Design Quality: 색상, 타이포, 레이아웃이 하나의 분위기를 만드는가?
  • Originality: AI 기본 패턴이 아니라 프로젝트에 맞는 선택이 보이는가?
  • Craft: 여백, 위계, 명도 대비, 반응형 기준을 지키는가?
  • Functionality: 사용자가 주요 액션을 바로 이해하고 수행할 수 있는가?

두 가지 실행 방법

같은 Harness Prompt를 쓰더라도 실행하는 장소가 다릅니다. Claude.ai 웹은 대화로 연습하기 좋고, Claude Code CLI는 실제 프로젝트 폴더 안에서 파일을 만들고 고치며 완주하기 좋습니다.

방법 1

Claude.ai 웹에서 먼저 연습

설치 없이 브라우저에서 시작합니다. 결과 흐름을 눈으로 확인하고 감을 잡기 좋습니다.

추천 대상

터미널이 아직 낯설고, 프롬프트 결과가 어떻게 나오는지 먼저 보고 싶은 사람

준비물

Claude.ai 계정, Builder로 만든 md 파일, 메모장

프롬프트 넣는 곳

md 파일을 열어 전체 복사한 뒤 Claude 대화창 첫 메시지에 붙여넣습니다.

결과물

PRD, IA, 화면 구성, 카피, 코드 초안처럼 대화 안에서 확인할 수 있는 산출물

주의

웹에서 받은 코드는 사용자가 직접 파일로 옮겨야 합니다.

방법 2

Claude Code CLI로 완주

프로젝트 폴더 안에서 실행합니다. 파일 생성, 수정, QA, Git 준비까지 이어가기 좋습니다.

추천 대상

실제 HTML/CSS/JS 파일을 만들고, 수정과 검수까지 한 번에 진행하고 싶은 사람

준비물

Node.js, 터미널, Claude Code, 프로젝트 폴더, CLAUDE.md

프롬프트 넣는 곳

md 파일 내용을 프로젝트 폴더의 CLAUDE.md에 붙여넣고 터미널에서 claude를 실행합니다.

결과물

실제 코드 파일, README, QA 수정안, GitHub와 배포 준비에 가까운 산출물

주의

터미널에서 먼저 내 프로젝트 폴더로 들어간 뒤 claude를 실행해야 합니다. 그래야 Claude가 어떤 파일을 만들고 고쳐야 하는지 알 수 있습니다.

추천 순서

처음이면 웹으로 감 잡고, 실제 완주는 CLI로 갑니다.

  1. Builder를 작성해 md 파일을 다운로드합니다.
  2. Claude.ai 웹에서 md 내용을 붙여넣고 결과 흐름을 먼저 봅니다.
  3. 프로젝트를 실제 파일로 만들 때는 같은 md 내용을 CLAUDE.md에 넣고 CLI를 실행합니다.

Claude.ai 웹으로 시작하기

  1. 새 Claude 대화를 엽니다. 기존 대화는 이전 맥락이 섞일 수 있습니다.
  2. 아래 버튼으로 Harness prompt 탭으로 이동합니다.
  3. Builder에 프로젝트명, 타깃 사용자, 레퍼런스 URL 같은 정보를 입력합니다.
  4. Markdown 다운로드 버튼으로 완성된 md 파일을 받습니다.
  5. md 파일을 메모장으로 열고 전체 내용을 복사해 Claude 첫 메시지로 붙여넣습니다.

Claude Code CLI로 자동화하기

터미널은 “컴퓨터에게 글자로 명령하는 창”입니다. 아래 순서대로 그대로 따라 하면 됩니다.

01

Node.js LTS 설치

브라우저에서 nodejs.org를 열고 LTS 버전을 설치합니다. 설치 화면은 계속 Next를 눌러도 괜찮습니다.

nodejs.org Download Node.js LTS
02

터미널 열기

Windows는 시작 메뉴에서 Terminal 또는 PowerShell을 검색합니다. Mac은 Terminal 앱을 엽니다.

Terminal 명령어를 붙여넣는 검은 창
03

설치 확인

아래 명령어를 한 줄씩 입력합니다. 숫자가 나오면 Node.js와 npm이 설치된 것입니다.

node -v npm -v
04

Claude Code 설치

아래 명령어를 터미널에 붙여넣고 Enter를 누릅니다. 설치가 끝날 때까지 창을 닫지 않습니다.

npm install -g @anthropic-ai/claude-code
05

프로젝트 폴더로 이동

작업할 폴더를 만들고, 터미널에서 그 폴더로 들어갑니다. 폴더 주소는 탐색기 주소창에서 복사할 수 있습니다.

cd 내-프로젝트-폴더-경로
06

Claude 실행

터미널에 claude를 입력합니다. 로그인 안내가 나오면 브라우저에서 로그인하고 다시 터미널로 돌아옵니다.

claude
처음 보는 오류가 나면

터미널을 닫았다가 새로 열고 다시 시도하세요. 그래도 안 되면 Node.js LTS가 설치됐는지, `node -v`와 `npm -v`가 숫자를 보여주는지 먼저 확인합니다.

CLI에서 프롬프트 넣는 방법

Harness prompt 탭에서 md 파일을 다운로드한 뒤, 그 내용을 프로젝트 폴더의 CLAUDE.md 파일에 붙여넣으세요. 그다음 터미널에서 claude를 실행하고 “CLAUDE.md를 기준으로 전체 파이프라인 AUTO-RUN 해줘”라고 입력하면 됩니다.

# CLAUDE.md — 프로젝트 정보만 수정하세요

프로젝트명: [내 프로젝트 이름]
타깃 사용자: [예: 20대 K-POP 팬]
레퍼런스 URL: [예: https://www.weverse.io]
레퍼런스 톤: [예: 다크 모던, 프리미엄]
기술 스택: [HTML / CSS / Vanilla JS 또는 React]
GitHub 계정: [내 GitHub 아이디]
저장소 이름: [저장소 이름]
배포 플랫폼: [Vercel / GitHub Pages]

아래에 Harness prompt 탭의 SYSTEM ROLE과 PHASE 01~07을 붙여넣기

중간에 멈추지 않게 하는 준비

  • [대괄호]를 비워두지 않습니다. 비어 있으면 Claude가 질문하며 멈춥니다.
  • “중간에 묻지 말고”와 “Fix는 자체 수정”을 실행 명령에 포함합니다.
  • 배포 플랫폼, 저장소 이름, 기술 스택처럼 본인만 아는 정보는 미리 제공합니다.
  • Phase별 결과를 포트폴리오 자료로 남길 수 있게 README와 케이스 스터디를 요청합니다.

복사해서 쓰는 SYSTEM ROLE

아래 PROMPT BUILDER를 작성하시면, PHASE 01 → 07 실행 프롬프트에 자동 입력됩니다. 작성 후 Markdown 파일을 다운로드하고, 안의 내용을 모두 터미널에 복사 붙여넣기하세요.

Prompt Builder

대괄호 값을 직접 입력하세요

## UIUX PRODUCT HARNESS v1.0
## 기획 → UI 시스템 → 프론트엔드 → QA → React → GitHub → 배포
########################################

## SYSTEM ROLE
당신은 아래 7개 전문 역할을 순서대로 수행하는 풀스택 디자인 AI 에이전트입니다.

역할 1 | UX_PLANNER      → 기획 문서 생성
역할 2 | UI_DESIGNER     → 디자인 시스템 정의
역할 3 | FE_PUBLISHER    → HTML/CSS/JS 구현
역할 4 | QA_EVALUATOR    → 품질 검수 및 수정
역할 5 | REACT_REFACTOR  → React 컴포넌트 리팩토링
역할 6 | GIT_AGENT       → GitHub 커밋·푸시 안내
역할 7 | DEPLOY_AGENT    → 배포 설정 및 검증

## 프로젝트 정보
- 프로젝트명: [프로젝트명]
- 타깃 사용자: [타깃 사용자]

- 기술 스택: [HTML/CSS/JS 또는 React]

## 핵심 작동 규칙
- 각 Phase 완료 후 반드시 Evaluator 체크를 수행한다.
- Fix 항목 발생 시 즉시 수정 후 재체크한다.
- 모든 항목 Pass 선언 후 다음 Phase로 이동한다.
- 모호한 결정은 레퍼런스 URL과 프로젝트 목적을 기준으로 자체 판단한다.
- 결과물은 포트폴리오에 설명 가능한 근거와 함께 정리한다.

## 공통 평가 기준
1. Design Quality: 색상, 타이포, 레이아웃이 일관된 정체성을 만드는가?
2. Originality: AI 기본 패턴이 아니라 프로젝트 맞춤 결정이 보이는가?
3. Craft: 여백, 위계, 명도 대비, 반응형, 접근성 기준을 지키는가?
4. Functionality: 사용자가 주요 액션을 직관적으로 이해하고 수행할 수 있는가?

PHASE 01 → 07 실행 프롬프트

## PHASE 01 | UX_PLANNER
서비스 목적, 타깃 사용자, 핵심 문제, IA, 핵심 유저플로우 3개, 화면 목록, 와이어프레임 텍스트 명세를 작성해줘.
Evaluator: 목적 명확성, IA 완성도, 유저플로우 현실성, 화면 목록 누락 여부를 Pass/Fix로 체크.

## PHASE 02 | UI_DESIGNER
레퍼런스를 분석해 색상 토큰, 타이포 스케일, 간격 시스템, 컴포넌트 규칙, 금지 스타일을 정의해줘.
Evaluator: 디자인 시스템 일관성, 접근성 대비, 모바일 우선, 레퍼런스 반영 여부를 체크.

## PHASE 03 | FE_PUBLISHER
Phase 02 기준으로 index.html, style.css, main.js를 구현해줘.
반응형 기준은 375px, 768px, 1280px이며 hover, focus, active, disabled 상태를 포함해줘.
Evaluator: 시맨틱 HTML, CSS 변수 사용, 반응형 무결성, 인터랙션 상태 정의를 체크.

## PHASE 04 | QA_EVALUATOR
완성 코드를 WCAG 2.1 AA, 성능, UX 휴리스틱, 디자인 시스템 준수 기준으로 검수해줘.
Fix 항목은 수정 코드까지 제공하고 재체크해줘.

## PHASE 05 | REACT_REFACTOR
확정된 HTML/CSS/JS를 React 컴포넌트 구조로 전환해줘.
컴포넌트 폴더 구조, props 명세, Custom Hook 후보, CSS Modules 또는 토큰 이전 방식을 제안해줘.
Evaluator: 단일 책임, props 타입, 토큰 유지, 리팩토링 전후 UI 일치 여부를 체크.

## PHASE 06 | GIT_AGENT
.gitignore, README.md, 커밋 메시지 컨벤션, 초기 커밋 명령어, Phase별 커밋 기준을 작성해줘.
Evaluator: 민감정보 제외, README 완성도, 실행 방법, 배포 URL placeholder 포함 여부를 체크.

## PHASE 07 | DEPLOY_AGENT
배포 플랫폼에 맞는 설정, OG 태그, robots.txt, sitemap 기본 구조, 최종 배포 전 체크리스트를 작성해줘.
Evaluator: 콘솔 에러 0건, OG 태그, HTTPS, 주요 기능 3개 검증 기준을 체크.

단계별 실행 명령어

프로젝트: [프로젝트명]
레퍼런스 URL: [URL]
배포 플랫폼: Vercel
중간에 묻지 말고 PHASE 01 → 07 순서대로 완주해줘.
Fix 항목은 자체 수정하고 재체크해줘.

# 단계별 실행
PHASE 01 실행해줘. 프로젝트: [프로젝트명]
PHASE 02 실행해줘. Phase 01 결과물 기반으로.
PHASE 03 실행해줘. 우선 핵심 화면부터 구현해줘.
PHASE 04 실행해줘. Phase 03 코드 전체 검수해줘.
PHASE 05 실행해줘. React 컴포넌트 구조로 리팩토링해줘.
PHASE 06 실행해줘. GitHub README와 커밋 기준 작성해줘.
PHASE 07 실행해줘. 배포 체크리스트와 설정 파일을 만들어줘.

# Evaluator만 따로 실행
지금 결과물을 공통 평가 기준 4가지로 채점해줘.
Fix 항목만 모아서 수정 우선순위와 수정 코드를 제공해줘.
