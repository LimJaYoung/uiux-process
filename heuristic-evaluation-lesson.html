사용성 평가

초급 → 중급
Overview

휴리스틱 평가란?

"Heuristic"은 경험에서 도출된 원칙을 뜻합니다. UX에서는 Jakob Nielsen의 10가지 원칙을 기준으로 전문가가 UI를 직접 탐색하며 문제를 찾아내는 방식입니다. 실제 사용자 없이도 핵심 사용성 문제를 빠르게 발견할 수 있습니다.

전문가가 수행

UX 전문가 3~5명이 독립적으로 UI를 탐색합니다. 사용자를 섭외할 필요가 없습니다.

빠르고 저비용

1~2일 안에 주요 문제를 발견할 수 있습니다. 초기 프로토타입 검토에 특히 유용합니다.

10가지 기준 사용

검증된 10개 원칙으로 평가하므로 평가자마다 기준이 달라지는 혼선이 없습니다.

심각도 분류

발견한 문제를 0~4등급으로 분류해 개선 우선순위를 명확하게 정할 수 있습니다.

언제 사용하나요?

프로토타입 완성 직후 / 사용자 테스트 전 사전 점검 / 경쟁 앱 분석 / 기존 서비스 개선 검토 시 활용합니다. 사용자 테스트와 함께 쓰면 더욱 강력한 인사이트를 얻을 수 있습니다.

Core Principles

Jakob Nielsen의 10가지 휴리스틱

각 항목을 클릭하면 설명, 사례, 평가 포인트를 확인할 수 있습니다.

H1
Visibility of System Status

시스템 상태의 가시성

시스템은 지금 무슨 일이 일어나고 있는지 항상 사용자에게 알려야 합니다.
버튼 클릭, 로딩, 저장, 전송 등 모든 행동에 적절한 피드백이 필요합니다.
피드백이 없으면 사용자는 시스템이 반응하는지 알 수 없어 혼란을 느낍니다.

나쁜 예

업로드 버튼을 눌렀는데 아무 반응이 없음 — 진행 중인지, 오류인지 알 수 없음

좋은 예

파일 업로드 시 진행률 바(%) 표시 + 완료 시 "✓ 저장되었습니다" 토스트 메시지

카카오톡 읽음 확인 '1' 토스 결제 완료 화면 배민 주문 단계 표시
평가 포인트
  1. 버튼 클릭 시 즉각적인 시각 피드백(색상 변화, 스피너)이 있는가?
  2. 시간이 걸리는 작업에서 로딩/진행 상태를 알 수 있는가?
  3. 작업 완료 또는 실패 결과를 명확하게 안내하는가?
H2
Match Between System and the Real World

시스템과 현실 세계의 일치

시스템은 사용자에게 익숙한 언어, 단어, 개념, 은유를 사용해야 합니다.
개발 전문 용어 대신 사용자 언어로 소통하고, 현실 세계의 관습을 따라야 합니다.
사용자가 이미 아는 개념을 UI에 적용하면 별도 학습 없이 직관적으로 사용할 수 있습니다.

나쁜 예

오류 메시지에 "HTTP 500 Internal Server Error" 그대로 표시

좋은 예

쇼핑몰 '장바구니' 아이콘 — 실제 장바구니 메타포로 설명 없이도 바로 이해

장바구니 아이콘 달력(Date Picker) UI 저장 = 플로피 디스크 아이콘
평가 포인트
  1. 버튼·메뉴 레이블이 사용자에게 익숙한 단어를 사용하는가?
  2. 아이콘이 일반적으로 통용되는 의미를 가지는가?
  3. 에러 메시지에 기술 용어나 오류 코드가 없는가?
H3
User Control and Freedom

사용자 통제권과 자유

사용자는 언제든 실수를 합니다. 원하지 않는 상태에서 쉽게 빠져나올 수 있는 비상구가 필요합니다.
실행 취소(Undo), 취소 버튼, 뒤로 가기 등이 항상 명확하게 제공되어야 합니다.
탈출구가 있으면 사용자는 심리적 안전감을 느끼고 서비스를 더 적극적으로 탐색합니다.

나쁜 예

긴 회원가입 폼에서 뒤로가기를 누르면 입력 내용이 모두 사라짐

좋은 예

Gmail 이메일 삭제 후 하단에 5초간 '실행 취소' 버튼 표시

Gmail 실행 취소 Figma Ctrl+Z 모달 X 버튼 / ESC
평가 포인트
  1. 모달·팝업에 닫기(X) 버튼이 명확하게 있는가?
  2. 다단계 플로우에서 이전 단계로 돌아갈 수 있는가?
  3. 비가역적 행동(삭제, 탈퇴) 전에 확인 단계 또는 취소 방법이 있는가?
H4
Consistency and Standards

일관성과 표준

같은 단어, 상황, 행동은 항상 같은 의미를 가져야 합니다.
앱 내 일관성(버튼 위치·색상·레이블 통일)과 플랫폼 표준(iOS/Android 가이드라인) 모두 중요합니다.
일관성이 높을수록 사용자의 학습 비용이 줄고 사용 속도가 빨라집니다.

나쁜 예

어떤 화면에서는 '확인' 버튼이 오른쪽, 다른 화면에서는 왼쪽에 위치

좋은 예

토스 — 앱 전체에서 Primary CTA는 항상 하단 중앙에 동일한 스타일로 배치

iOS HIG 뒤로가기 좌상단 토스 디자인 시스템 Material Design
평가 포인트
  1. 같은 기능의 버튼이 화면마다 동일한 위치·색·레이블을 유지하는가?
  2. 같은 행동에 다른 용어를 혼용하지 않는가? (예: '삭제' vs '지우기' 혼용)
  3. 플랫폼(iOS/Android) 관례를 따르는가?
H5
Error Prevention

오류 예방

좋은 오류 메시지도 중요하지만, 처음부터 오류가 발생하지 않도록 설계하는 것이 더 중요합니다.
실수하기 어려운 UI가 가장 좋은 UI입니다.
오류 발생 조건을 제거하거나, 행동 전에 사용자가 확인하도록 유도해야 합니다.

나쁜 예

폼을 제출한 후에야 이메일 형식 오류를 알려줌 — 입력 중 검사가 없음

좋은 예

날짜 피커에서 선택 불가능한 과거 날짜를 회색으로 비활성화 처리

인라인 유효성 검사 비밀번호 강도 표시기 삭제 확인 다이얼로그
평가 포인트
  1. 입력 필드에 실시간 유효성 검사(인라인 에러)가 있는가?
  2. 선택 불가능한 옵션이 시각적으로 비활성화되어 있는가?
  3. 비가역적 행동(삭제, 결제 확정) 전에 확인 단계가 있는가?
H6
Recognition Rather Than Recall

기억보다 인식

사용자가 기억해야 하는 부담을 최소화해야 합니다.
보기를 보고 '아, 맞아!' 하는 Recognition이, 아무 단서 없이 기억해야 하는 Recall보다 훨씬 쉽습니다.
필요한 옵션과 정보는 눈에 보이게 제공하고, 시스템이 사용자의 이전 행동을 기억해야 합니다.

나쁜 예

검색창에 최근 검색어나 자동완성이 없어 매번 처음부터 직접 입력해야 함

좋은 예

배달의민족 — 최근 주문한 메뉴 상단 노출로 재주문 원클릭 가능

쿠팡 최근 검색어 배민 최근 주문 비밀번호 보기 토글
평가 포인트
  1. 이전에 했던 행동(최근 검색, 최근 방문)이 화면에 표시되는가?
  2. 선택 가능한 옵션이 드롭다운·체크박스 등 보이는 형태로 제공되는가?
  3. 아이콘 단독이 아닌 텍스트 레이블도 함께 제공하는가?
H7
Flexibility and Efficiency of Use

유연성과 효율성

초보자에게는 친절한 가이드가, 숙련자에게는 빠른 경로(단축키, 즐겨찾기 등)가 필요합니다.
두 유형의 사용자를 모두 배려하는 설계가 중요합니다.
숙련자를 위한 기능이 초보자의 경험을 방해하지 않도록 설계해야 합니다.

나쁜 예

반복 주문 기능이 없어 동일 상품을 매번 처음부터 검색·선택·담기를 반복해야 함

좋은 예

Figma 단축키 — R(사각형), T(텍스트), V(선택) 으로 도구 전환 즉시 가능

Figma 키보드 단축키 배민 단골가게 은행 자주 쓰는 계좌
평가 포인트
  1. 자주 쓰는 기능에 빠르게 접근할 수 있는 바로가기·즐겨찾기가 있는가?
  2. 반복 작업을 한 번에 처리하는 기능이 있는가? (재주문, 일괄 처리 등)
  3. 숙련자용 기능이 초보자 경험을 방해하지 않는가?
H8
Aesthetic and Minimalist Design

심미적이고 미니멀한 디자인

화면에는 사용자 목표에 꼭 필요한 정보만 있어야 합니다.
불필요한 요소는 중요한 정보의 가시성을 낮추고 인지 부담을 높입니다.
미니멀 디자인은 '예쁘게'가 아니라 '불필요한 것을 제거하는 것'입니다.

나쁜 예

앱 실행 시 광고 팝업 3개가 연속으로 표시되어 핵심 콘텐츠 진입을 막음

좋은 예

토스 각 화면 — 하나의 핵심 행동(CTA)만 존재, 불필요한 메뉴·배너 없음

토스 단일 CTA Apple 제품 페이지 Google 홈 화면
평가 포인트
  1. 한 화면에 CTA가 1~2개로 집중되어 있는가?
  2. 화면의 모든 요소가 현재 사용자 목표에 필요한가?
  3. 광고·팝업이 핵심 콘텐츠를 가리지 않는가?
H9
Help Users Recognize, Diagnose, and Recover from Errors

오류 인식·진단·복구 지원

오류 메시지는 ① 무슨 문제인지 / ② 왜 발생했는지 / ③ 어떻게 해결하는지를 알려야 합니다.
기술 용어나 오류 코드 없이 사용자 언어로 작성해야 합니다.
명확한 오류 안내는 사용자를 이탈 없이 목표 달성으로 다시 안내합니다.

나쁜 예

"오류가 발생했습니다." — 무슨 오류인지, 어떻게 해결하는지 전혀 알 수 없음

좋은 예

"이메일 또는 비밀번호가 일치하지 않습니다. [비밀번호 찾기]" — 원인 + 해결 링크 제공

인라인 폼 에러 메시지 404 페이지 + 홈으로 네트워크 오류 + 재시도
평가 포인트
  1. 오류 메시지에 기술 용어·오류 코드가 없는가?
  2. 어떤 필드·항목이 문제인지 구체적으로 알려주는가?
  3. 오류 이후 다음 행동(재시도, 수정, 관련 링크)을 안내하는가?
H10
Help and Documentation

도움말과 문서화

도움말 없이도 쓸 수 있는 UI가 이상적이지만, 복잡한 기능에는 맥락에 맞는 도움말이 필요합니다.
도움말은 쉽게 찾을 수 있어야 하고, 작업 중심으로 구체적인 단계를 안내해야 합니다.
방대한 매뉴얼보다 지금 이 순간 바로 쓸 수 있는 인라인 가이드가 효과적입니다.

나쁜 예

복잡한 설정 화면에 아무런 설명이 없고, 도움말 링크를 클릭하면 PDF 매뉴얼이 다운로드됨

좋은 예

입력 필드 아래 "예: 사업자번호는 000-00-00000 형식입니다" helper text 즉시 표시

온보딩 툴팁 인라인 Helper Text Notion 도움말 센터
평가 포인트
  1. 복잡한 기능 옆에 인라인 설명(helper text, 툴팁)이 있는가?
  2. 처음 사용자를 위한 온보딩 가이드가 제공되는가?
  3. 도움말·FAQ를 쉽게 찾을 수 있고 검색이 가능한가?
Severity Rating

심각도 평가 척도 (0~4)

발견한 문제를 5단계로 분류해 개선 우선순위를 결정합니다. 수치화해야 팀이 행동할 수 있습니다.

등급 0

문제 없음

다른 평가자가 지적했으나 실제로는 사용성 문제가 아닌 경우입니다.

수정 불필요
등급 1

표면적 문제

발생 빈도 낮고 사용자가 쉽게 우회할 수 있습니다. 여유 있을 때 개선합니다.

여유 시 수정
등급 2

중간 심각도

불편하지만 해결 방법을 찾을 수 있습니다. 다음 릴리즈에 반드시 포함합니다.

다음 릴리즈 수정
등급 3

심각한 문제

사용자가 해결 방법을 찾기 어렵거나 포기 가능성이 높습니다. 출시 전 필수 수정합니다.

출시 전 필수 수정
등급 4

치명적 문제

핵심 태스크를 완료할 수 없게 만드는 차단(Blocker) 문제입니다.

즉시 수정 필수

심각도 결정 3요소

빈도 — 얼마나 많은 사용자가 자주 겪는가?  영향 — 문제를 극복하기 얼마나 어려운가?  지속성 — 한 번 겪으면 계속 방해받는가?

Evaluation Process

평가 수행 5단계

처음 수행하는 팀도 이 순서를 따르면 체계적으로 평가할 수 있습니다.

01
01

평가 계획

  • 평가할 화면·플로우 선정
  • 평가자 3~5명 선정
  • 10가지 원칙 사전 공유
  • 평가 시트(템플릿) 준비
02
02

개별 평가

  • 평가자 간 교류 없이 독립 수행
  • 10원칙 하나씩 대조하며 탐색
  • 위반 원칙(H1~H10)과 함께 기록
  • 화면 캡처 병행
03
03

결과 취합

  • 전체 발견 사항 스프레드시트에 병합
  • 중복 항목 통합
  • 화면별·원칙별로 분류
  • 이 단계에서 처음으로 공유
04
04

심각도 평가

  • 각 항목에 0~4 심각도 부여

평가자는 몇 명이 적당한가?

Nielsen 연구에 따르면 1명은 약 35%, 5명은 약 75~85%의 문제를 발견합니다. 6명 이상부터는 발견 비율이 크게 늘지 않아 실무에서는 3~5명이 적정입니다.
