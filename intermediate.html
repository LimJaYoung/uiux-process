01. 아이템 선정

취업을 위한 포트폴리오 첫 단계는 “무엇을 만들 것인가”입니다. 좋아하는 브랜드보다 결과물의 시각적 상한선이 높은 브랜드를 골라야 합니다. K-브랜드 중에서도 리뉴얼 명분이 분명하고, 인터랙티브 웹으로 풀었을 때 임팩트가 큰 브랜드가 적합합니다.

  • Before & After의 결과가 명확할 수 있는가
  • 현재 웹사이트의 문제점이 명확해 리뉴얼 명분이 생기는가
  • 이미지, 영상, 공간, 제품, 라이프스타일 등 비주얼 소스가 풍부한가
  • 스크롤 애니메이션, 모션 타이포, 섹션 전환이 자연스럽게 어울리는가
  • 최상급 퀄리티의 레퍼런스 자료가 충분한가
너는 UIUX 기획자이자 포트폴리오 디렉터야.

취업용 UIUX 풀스택 포트폴리오를 제작하기 위해
K-브랜드 웹사이트 리뉴얼 반응형 웹 포트폴리오를 만들려고 해.

목표:

- Figma 시안과 HTML/CSS/JavaScript 구현까지 연결
- 면접관이 첫 화면에서 시각적 임팩트를 느낄 수 있는 결과물 만들기

03-A. 참고 레퍼런스 제시

이 단계에서는 바로 디자인 시스템을 추출하지 않습니다. 먼저 리뉴얼 방향에 맞는 최상급 레퍼런스 후보를 여러 개 제시하고, 사용자가 그중 하나를 모태 레퍼런스로 선택합니다. 선택이 끝난 뒤 다음 단계에서 해당 사이트의 폰트, 여백, 그리드, 이미지 비율을 정밀 추출합니다.

  • 모태 레퍼런스 후보 5~8개를 먼저 제시한다.
  • 각 후보의 레이아웃, 타이포, 이미지, 모션, 구현 난이도를 비교한다.
  • 사용자가 후보 중 1개를 선택할 수 있도록 장단점을 명확히 정리한다.
  • 선택된 1개 사이트만 다음 단계에서 디자인 시스템 추출 대상으로 사용한다.

레퍼런스 후보 선정 원칙

레퍼런스는 사용자가 충분히 비교·판단할 수 있도록 최소 7개의 후보를 제시하며, 모든 레퍼런스는 실제 확인 가능한 URL을 함께 제공합니다.

사용자는 제시된 후보를 검토한 후 최종 모태 레퍼런스 1개를 직접 선정합니다. 선정된 모태 레퍼런스는 프로젝트의 전체 레이아웃, 타이포그래피, 그리드 시스템, 콘텐츠 구성의 기준이 되며, 이후 디자인 방향성 역시 이를 중심으로 전개합니다.

모태 레퍼런스 선정은 프로젝트의 최종 비주얼 퀄리티를 결정하는 핵심 과정입니다. 따라서 사용자가 원하는 방향성과 수준에 부합하는 레퍼런스를 찾을 때까지 후보 탐색, 비교, 교체 과정을 반복적으로 진행합니다. 충분한 검토 없이 레퍼런스를 확정하지 않으며, 사용자가 납득할 수 있는 기준과 근거를 바탕으로 최종 레퍼런스를 선정합니다.

서브 레퍼런스는 기본적으로 제공하지 않으며, 사용자가 추가 요청한 경우에만 히어로 섹션, 카드 UI, 모션, CTA, 인터랙션 등 특정 요소에 한하여 별도로 제안합니다. 단, 서브 레퍼런스는 부분 요소 참고용으로만 활용하며 전체 디자인 구조의 기준은 항상 모태 레퍼런스를 유지합니다.

후보 선정 기준:
- 전체 레이아웃과 타이포 시스템을 따라가기 좋고, 실제 확인 가능한 링크를 연결할 수 있는 사이트
- 폰트, 여백, 이미지 비율, 섹션 리듬이 명확한 사이트
- K-브랜드 리뉴얼에 적용했을 때 Before & After 차이가 강하게 보일 수 있는 사이트
- Awwwards, Godly, Land-book, Framer, Webflow, 글로벌 브랜드 사이트처럼 퀄리티가 높은 사례 우선
- 같은 업종 레퍼런스와 타 업종 레퍼런스를 섞되, 최종 모태 레퍼런스는 1개만 선택할 수 있게 정리
- 모든 후보는 실제 확인 가능한 URL을 함께 제공

03-B. 레퍼런스 디자인 시스템 추출

가장 중요한 원칙:
- 절대 감으로 추정하지 말 것

- 모든 디자인 시스템 값은 반드시 선택한 모태 레퍼런스에서 확인한 근거를 기준으로 정리할 것
- 모태 레퍼런스에서 확인하지 못한 값은 비슷하게 만들어내지 말고 "확인 필요"로 남길 것
- "우리 브랜드에 적용 방식"도 모태 레퍼런스의 구조와 수치를 출발점으로 작성할 것

반드시 지킬 것:

- 가능하면 브라우저 개발자도구, Playwright, CSS 검사, 스크린샷 측정 중 가능한 방법으로 실제 값을 확인할 것
- 실제 값을 확인할 수 없는 항목은 추측하지 말고 "확인 필요"라고 표시하고, 확인 방법을 적을 것
- "고급스럽게", "미니멀하게", "여백을 넉넉하게"처럼 감상형 표현으로 끝내지 말 것
- 폰트명, 폰트 두께, 자간, 줄간격, 폰트 크기, 섹션 여백, 카드 내부 여백을 수치로 작성할 것
- 각 추출값마다 확인 위치/근거를 반드시 작성하고, 근거가 없는 값은 추출값으로 확정하지 말 것
- 모태 레퍼런스에 접근할 수 없다면 분석을 진행하지 말고, 원본 페이지 URL 또는 모태 이미지 URL을 다시 입력해 달라고 요청할 것

적용 설계에서 반드시 해야 할 것

모태 레퍼런스는 분위기 참고 자료가 아니라 실제 설계 기준입니다. index 페이지 전체를 캡처해 Figma에 넣고, 섹션 단위로 구조를 분해한 뒤 우리 브랜드의 콘텐츠와 리뉴얼 목표에 맞게 적용 방법을 제안해야 합니다.

  • Figma에 바로 그리기 전에 프레임, 그리드, 컬러, 텍스트, 네이밍 기준을 문서로 먼저 정리한다.
  • Figma 파일 URL을 Claude/Codex에 붙여넣고 MCP가 현재 파일과 프레임을 읽을 수 있는지 확인한다.
  • 확정된 요구서를 기준으로 Figma에 디자인 시스템을 등록하고, 이후 컴포넌트와 페이지 시안을 제작한다.
  • 바이브코딩 전 레이어명, 스타일 연결, 컴포넌트 구조, 모바일 대응까지 점검한다.

04-C. Figma에 디자인 시스템 등록
04-A 요구서를 기준으로 Figma MCP를 통해 프레임, 그리드, 컬러 스타일, 텍스트 스타일을 등록합니다.

04-D. 컴포넌트 제작

Figma MCP로 전체 시안을 한 번에 제작하게 하면 구조는 만들어질 수 있지만, 타이포 크기, 여백, 이미지 비율, 섹션 리듬이 무너지면서 MVP 수준으로 나올 가능성이 높습니다. 따라서 Playwright로 모태 레퍼런스를 먼저 확인하고, 그 분석 결과를 바탕으로 Figma MCP에서 섹션별로 제작과 검수를 반복해야 합니다.

1. Playwright 브라우징으로 모태 레퍼런스의 데스크톱/모바일 화면을 확인한다

5. Hero 섹션만 먼저 제작하고 비주얼 퀄리티를 확인한다
6. 피드백을 반영한 뒤 나머지 섹션을 순차적으로 제작한다
7. 마지막에 모바일 프레임을 제작하고 데스크톱과 같은 디자인 시스템을 공유하는지 점검한다

- 섹션 구성은 카피 전략의 메시지 흐름과 연결되게 설계할 것
- Hero 섹션을 만들기 전에 모태 레퍼런스 원본값과 적용값을 표로 다시 요약할 것
- 단, 브랜드명, 콘텐츠, 컬러, 이미지, 메시지는 리뉴얼 대상 브랜드에 맞게 재해석할 것
- 모태 레퍼런스를 보지 못했거나 접근할 수 없다면 시안 제작을 시작하지 말고, 먼저 접근 불가 이유와 필요한 자료를 알려줄 것

제작 순서:
1. 메인 페이지 Hero 섹션
   - 첫인상을 결정하는 핵심 비주얼
   - Before & After 차이가 강하게 느껴지는 타이포와 이미지 구조
2. 메인 페이지 나머지 섹션
   - 브랜드와 서비스 종류에 맞게 핵심 섹션을 재구성
   - 예: Brand Story / Product / Service / Benefit / Review / Store / CTA
   - 헬스장, 식품, 뷰티, 공간, 패션 등 업종에 따라 필요한 섹션명을 다르게 설계
3. 서브 페이지
   - 브랜드와 서비스 구조에 맞는 상세 페이지 구성
   - 예: Brand / Product / Service / Store / Membership / Contact
4. 모바일 프레임
   - Hero, GNB, CTA, 카드, 지점찾기 흐름 우선

시안 원칙:
- 모태 레퍼런스 주소와 추출된 디자인 토큰을 기준으로 전체 레이아웃 리듬 통일
- 섹션별 보조 레퍼런스는 부분 요소만 참고
- 타이포그래피, 이미지, 여백, 컬러, CTA 스타일을 하나의 시스템처럼 유지
- 등록된 컬러 스타일, 텍스트 스타일, 컴포넌트를 우선 사용
- 등록된 spacing/layout token 밖으로 임의 확장하지 말 것
- 구현 단계에서 HTML/CSS/JavaScript로 재현 가능한 구조로 정리

각 섹션별로 아래 표를 작성하며 작업해줘.

| 페이지 | 섹션 | 목적 | 모태 레퍼런스에서 가져온 구조/수치 | 적용한 Figma 스타일/토큰 | 핵심 비주얼 | 인터랙션 아이디어 | 모바일 대응 |
