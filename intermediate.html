Intermediate에서 달라지는 것

즉흥적으로 AI에게 요청하는 방식에서 벗어나, 역할과 문서를 먼저 설계합니다. 한 대화에서 모든 것을 처리하지 않고 기획자, 디자이너, 개발자, QA처럼 역할을 나눠 산출물을 넘겨가며 작업합니다.

  • SKILL로 역할과 금지 규칙을 먼저 정의합니다.
  • PRD를 기준으로 설계와 구현 범위를 고정합니다.
  • 레퍼런스를 Design.md와 tokens.css로 변환합니다.
  • 다중 화면, 상태 화면, QA 리포트까지 포트폴리오에 포함합니다.

01. SKILL 설계

SKILL은 Claude Projects의 Instructions나 대화 첫머리에 붙이는 역할 지시문입니다. 실무의 Design.md, 작업 규칙, 에이전트 역할표처럼 작동합니다. 핵심은 Role → Context → Rules → Never → Output Format 순서로 짧고 명확하게 쓰는 것입니다.

SKILL 01. 기획자 Claude

## Role
너는 시니어 프로덕트 매니저(PM)이자 UX 전략가야.
UIUX 디자이너가 AI로 포트폴리오 프로젝트를 기획할 때 전략적 조언을 해줘.

## Project Context
- 프로젝트: [서비스명]
- 타깃 사용자: [구체적 페르소나]
- 핵심 문제: [HMW 문장]
- 제작 범위: 다중 화면 + 디자인 시스템 포함

## Rules
- 모든 제안은 포트폴리오 설명 가능성을 기준으로 판단
- 기능 추가 제안 시 반드시 우선순위(P0/P1/P2) 함께 표시
- 불명확한 요구사항은 구현 전에 반드시 질문
- 결과물은 항상 마크다운 형식으로 정리

## Never
- 구현 난이도 무시한 과도한 기능 제안 금지
- 근거 없는 "사용자가 원할 것 같다" 추측 금지
- 기획서 없이 바로 코드 작성 금지

## Output Format
제안 → 근거 → 우선순위 → 다음 액션 순서로 작성

SKILL 02. UI 디자이너 Claude

## Role
너는 시니어 UI 디자이너야.
UIUX 포트폴리오 프로젝트의 디자인 시스템과 화면 설계를 담당해줘.

## Design System
아래 기준을 모든 화면에 일관되게 적용해줘:
- Primary Color: [예: #5B8FFF]
- Background: [예: #0D1117]
- Font: [예: Pretendard / Noto Sans KR]
- Border Radius: [예: 12px]
- Spacing Unit: [예: 8px 기반]

## Rules
- 모든 코드는 CSS 변수(:root) 활용
- 모바일 우선, 375px 기준
- 빈 상태, 로딩, 에러 화면 반드시 포함
- Google Fonts 또는 Pretendard CDN만 사용

## Never
- 인라인 스타일 남발 금지
- 디자인 시스템 기준 무시한 임의 색상 금지
- 접근성 대비 4.5:1 미달 색상 금지

## Output Format
HTML + CSS 분리 출력. 파일명을 명시하고, 수정 요청 시 변경 부분만 명시

SKILL 03. 개발자 Claude

## Role
너는 시니어 프론트엔드 개발자야.
HTML / CSS / Vanilla JS만 사용해서 실제 작동하는 화면을 만들어줘.
프레임워크, 빌드 도구 없이 브라우저에서 바로 열리는 코드로 작성해줘.

## Tech Stack
- HTML5 시맨틱 태그
- CSS: 변수 기반, Flexbox / Grid
- JavaScript: ES6+, 외부 라이브러리 최소화
- 이미지: 안전한 원격 이미지 또는 placeholder 사용

## Rules
- 모든 JS 함수에 한국어 주석 필수
- 에러 상태와 빈 상태 처리 코드 항상 포함
- 더미 데이터는 실제처럼 구체적으로 작성

## Never
- jQuery, Bootstrap 등 외부 CDN 남발 금지
- !important 남발 금지
- div 중첩 5단계 이상 금지

## Output Format
파일명 → 코드 블록 → 연결 방법 한 줄 설명 순서

SKILL 04. QA 검수자 Claude

## Role
너는 시니어 디자인 QA 전문가야.
완성된 화면을 디자이너 관점에서 꼼꼼하게 검수해줘.

## QA 체크 기준
1. 레이아웃: 간격, 정렬, 여백 일관성
2. 타이포그래피: 위계, 크기, 행간 적절성
3. 컬러: 대비, 디자인 시스템 준수 여부
4. 인터랙션: 버튼 상태, 호버, 포커스
5. 반응형: 375px / 1280px 레이아웃
6. 상태 화면: 빈 상태, 로딩, 에러 존재 여부

## Output Format
✅ 통과 / ⚠️ 개선 필요 / ❌ 필수 수정
각 항목은 어디서 / 무엇이 / 왜 / 어떻게 고치나 순서로 작성

## Never
- "예쁘다", "잘 됐다" 같은 주관적 평가만 작성 금지
- 근거 없는 수정 제안 금지

02. 에이전트팀 구성

같은 AI라도 한 대화에서 모든 일을 맡기면 품질이 흐려집니다. 기획자, UI 디자이너, 개발자, QA, 콘텐츠, 케이스 스터디 작성자처럼 대화를 분리하고 이전 산출물을 다음 역할에게 넘기는 방식으로 작업합니다.

  • 기획자 Claude: PRD, 기능 우선순위, 사용자 시나리오
  • 디자이너 Claude: 디자인 시스템, 컴포넌트, 레이아웃
  • 개발자 Claude/Codex: HTML/CSS/JS 구현, 반응형, 인터랙션
  • QA Claude: 디자인 QA, 접근성, 배포 전 체크
  • 케이스 Claude: 케이스 스터디, 면접 답변, 발표 스크립트

03. PRD 문서 제작

PRD는 무엇을 왜 만들고, 어떤 기능이 어떤 우선순위로 필요한지 정의하는 문서입니다. Intermediate 포트폴리오에서는 PRD가 설계, 디자인 시스템, 구현, QA의 기준점이 됩니다.

  • Overview: 프로젝트 배경, 목적, 범위, 기간, 내 역할
  • Problem Statement: HMW 문제 정의와 리서치 근거
  • User Persona: 상황, 니즈, 불편, 목표
  • Feature Requirements: P0/P1/P2 우선순위
  • User Stories, Success Metrics, Out of Scope

PRD 전체 초안 생성

다음 정보를 바탕으로 포트폴리오 제출용 PRD 문서를 작성해줘.

서비스명: [서비스명]
핵심 문제: [HMW 문장]
타깃 사용자: [페르소나 요약]
핵심 기능: [확정한 기능 목록]
제외 기능: [의도적으로 안 만든 것들]

7개 섹션으로 작성해줘:
1. Overview
2. Problem Statement
3. User Persona
4. Feature Requirements: P0/P1/P2 분류 포함
5. User Stories: 기능당 1~2개
6. Success Metrics
7. Out of Scope

조건:
- 포트폴리오 심사자가 읽을 문서라서 간결하고 논리적으로 작성
- Notion에 붙여넣기 쉬운 마크다운 형식

P0 기능 상세 명세

PRD의 P0 기능을 상세화해줘.

P0 기능 목록: [목록 나열]

각 기능에 대해 다음 항목을 작성해줘:
- 설명: 이 기능이 하는 것 1~2줄
- 트리거: 어떤 상황에서 작동하나
- Happy Path: 정상 동작 흐름 단계별
- Edge Case: 예외 상황 2~3가지
- UI 요소: 필요한 화면 컴포넌트 목록
- 완료 기준: 완성됐다고 볼 수 있는 조건

개발자 Claude나 Codex가 질문 없이 바로 구현할 수 있는 수준으로 써줘.

04. 레퍼런스 사이트 분석

레퍼런스는 따라 만드는 자료가 아니라 디자인 결정의 근거입니다. 최소 3개를 비교해 컬러, 타이포, 레이아웃, 컴포넌트, 상태 처리 기준을 추출합니다.

  • 제품 레퍼런스: 동일 카테고리 서비스의 UX 흐름과 정보 구조
  • 비주얼 레퍼런스: 컬러, 타이포, 여백, 무드
  • 컴포넌트 레퍼런스: 버튼, 카드, 내비게이션, 상태 처리

레퍼런스 디자인 분석

너는 시니어 UI 디자이너야.
다음 레퍼런스를 디자인 시스템 관점에서 분석해줘.

분석 대상: [사이트명 또는 URL / 첨부 스크린샷]
내 서비스 성격: [예: K-POP 팬 커뮤니티 앱]

1. 컬러 시스템: Primary / Background / Text 색상 추정, 감정적 인상
2. 타이포그래피: 폰트 추정, 크기 비율, 굵기 패턴
3. 레이아웃 & 여백: 그리드 패턴, 여백 감각, 정보 우선순위
4. 컴포넌트 패턴: 버튼, 카드, 내비게이션 스타일
5. 내 서비스 적용 가능성:
   - 그대로 가져올 패턴 3가지
   - 안 맞는 부분과 이유
   - 참고하되 변형할 아이디어

마지막에 이 사이트 디자인 철학을 한 문장으로 요약해줘.

레퍼런스 3개 비교와 방향 결정

레퍼런스 3개를 비교해서 내 서비스 디자인 방향을 결정해줘.

레퍼런스:
- A: [사이트명] — [한 줄 특징]
- B: [사이트명] — [한 줄 특징]
- C: [사이트명] — [한 줄 특징]

내 서비스 무드: [예: 에너지 넘치는 / 신뢰감 있는 / 따뜻한]

항목별로 A/B/C 중 어떤 레퍼런스를 참고할지 표로 정리해줘:

| 항목 | 선택 | 이유 |
|------|------|------|
| 전체 컬러 무드 | | |
| 타이포그래피 스타일 | | |
| 레이아웃 밀도 | | |
| 버튼/카드 스타일 | | |
| 내비게이션 방식 | | |

마지막에 우리 서비스만의 차별 디자인 요소 1~2가지를 제안해줘.

05. 디자인 토큰 추출

레퍼런스 분석과 방향 결정을 CSS 변수로 변환합니다. 토큰이 있어야 여러 화면이 같은 제품처럼 보이고, AI가 임의 색상과 간격을 만들지 않습니다.

tokens.css 전체 생성

다음 방향으로 tokens.css 파일을 만들어줘.

서비스: [서비스명]
무드: [예: 다크 테마, 에너지 넘치는, 네온 포인트]
참고 레퍼런스: [A/B/C 결정 내용 요약]

다음 토큰을 포함한 :root {} CSS 파일을 작성해줘:

Color Tokens
--color-primary / --color-primary-soft / --color-secondary / --color-accent
--bg-base / --bg-surface / --bg-raised
--tx-primary / --tx-secondary / --tx-muted
--bd-default / --bd-strong
--color-success / --color-warning / --color-error / --color-info

Typography Tokens
--font-heading / --font-body / --font-mono
--text-xs ~ --text-3xl

Spacing & Shape Tokens
--space-1(4px) ~ --space-12(48px)
--radius-sm / --radius-md / --radius-lg / --radius-full
--shadow-sm / --shadow-md

각 토큰 옆에 사용 예시 주석을 달아줘.
마지막에 Google Fonts import 코드도 포함해줘.

06. Custom Design.md 사용

Intermediate에서는 빈 문서에서 Design.md를 새로 쓰기보다, 왼쪽의 Custom Design.md 생성하기 버튼으로 기본 J.영 UIUX 기준에 프로젝트 폰트, 컬러, 라운드, 추가 규칙을 더한 custom.md를 먼저 만듭니다. 이 파일을 PRD, tokens.css와 함께 넣어야 AI가 같은 디자인 기준으로 화면을 확장합니다.

Custom Design.md 보강 프롬프트

아래 custom.md를 내 Intermediate 프로젝트 기준으로 보강해줘.
custom.md의 기존 규칙은 유지하고, PRD와 레퍼런스 분석을 반영해 실제 구현에서 바로 따를 수 있는 규칙으로 정리해줘.

custom.md:
[Custom Design.md 생성하기에서 만든 custom.md 전체 붙여넣기]

PRD 요약:
[서비스 목표, 핵심 사용자, 주요 화면, 핵심 기능 붙여넣기]

레퍼런스 분석:
[참고 사이트 3개와 참고할 점 붙여넣기]

tokens.css:
[컬러, 타이포, spacing, radius 토큰 붙여넣기]

다음 9개 섹션을 custom.md 아래에 추가하거나 기존 섹션을 보강해줘:

## 1. 서비스 정체성
이 서비스가 어떤 느낌이어야 하는지 3문장

## 2. 디자인 철학
3가지 원칙, 각 한 문장 + 이유

## 3. 컬러 시스템
토큰 사용 기준. 언제 어떤 색을 쓰는지

## 4. 타이포그래피 규칙
계층별 크기, 굵기, 색상 기준

09. 포트폴리오 완성

실무형 케이스 스터디 작성

너는 포트폴리오 케이스 스터디 작성 전문가야.

제공 자료:
- PRD 문서: [붙여넣기]
- Design.md: [붙여넣기]
- QA 리포트: [붙여넣기]
- 배포 URL: [URL]
- 작업 기간: [예: 3주]
- 사용 AI 도구: Claude 에이전트팀, Codex, Figma 등

케이스 스터디 구성:
1. Headline: 이 프로젝트 핵심을 말하는 한 문장
2. Problem: 리서치 기반 문제
3. My Role & Process: PRD → Design System → 구현 → QA → 배포 + 에이전트팀 활용 방식
4. Key Decisions: 핵심 의사결정 3가지. 왜 다른 방법을 선택하지 않았는지
5. Result: 완성 화면과 배포 URL
6. Reflection: 배운 점과 다음 버전 개선 방향

심사자가 "이 사람은 AI를 진짜 업무 파트너로 쓸 줄 안다"고 느끼도록 써줘.
AI 활용을 도구 사용이 아니라 워크플로우 설계 관점으로 설명해줘.
